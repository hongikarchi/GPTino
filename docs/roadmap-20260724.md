## #1 서드파티 플러그인 카탈로그 확장 + 플러그인 토글 (우선순위 1)

- GH 컴포넌트 서버(object proxy) 열거를 카탈로그에 편입 — 서드파티 컴포넌트 포함(이름/카테고리/
  입출력/GUID). 기존 GUID 라이브 검증 경로에 그대로 얹힘.
- 세션 설정에 **플러그인별 on/off 토글**. 모델 컨텍스트에는 설치+활성 플러그인의 컴포넌트만
  화이트리스트로 주입 (Raven의 외부 플러그인 능력의 실체가 이것 — 학습 데이터 아님).
- 검증: 플러그인별 벤치 2~3라운드로 friction 발견 → 서버/쿡북으로 소거 (기존 벤치 루프 그대로).
  1차 대상 **Karamba, Ladybug**.

**역할 분담 원칙 (확정)**:
- 솔버 도메인(구조=Karamba FEA, 환경=Ladybug/EnergyPlus, 물리=Kangaroo) → **네이티브 컴포넌트 배선**.
  C# 재구현 금지(거짓성공 위험 + 책임 소재).
- 기하 유틸리티 → **C# 스크립트** (현행 유지).
- 판단 기준: *결과를 사람이 이어받아 만지면 컴포넌트, 코드가 루프 안에서 소비하고 숫자만 남기면
  라이브러리.*
- 예외: **최적화는 코드/에이전트 루프 우위** — Galapagos는 수동 UI라 에이전트 구동 불가.
  브로커 루프(슬라이더 변경 → inspectOutputs로 적합도)나 C# GA가 자동화에 맞음.

## #2 검증 신뢰도 라벨 (우선순위 2)

- **서버가 결정론적으로 계산** — 모델 자기신고 금지(자기 채점 = 거짓성공 재발).
  실행된 검증에서 파생: 샘플값 실측(inspectOutputs) = `high` / 솔브만 통과 = `medium` /
  미검증 = `low`.
- 단계적 도입:
  1. 잡 결과(applied/committed의 outputs 요약)에 필드로 노출 + 패널 리비전별 표시 — **정보만**.
  2. 벤치로 효과 확인 후 하우스룰 소프트 정책 검토("low 상태 커밋 전 재고" 수준).
- **하드 게이트 금지** — "모델이 알 수 없는 것을 요구하는 검증기" 클래스 재발명 위험
  (wireify 수렴에서 소거한 바로 그 부류).

## #3 사내 정의 스니펫 블랙박스 등록 (회사 배포 시점에 맞춰)

- 사용자가 사내 표준 .ghx/클러스터를 **한 줄 설명**과 함께 등록 → 카탈로그 `kind=snippet` →
  에이전트는 내부를 파싱하지 않고 설명 기반으로 통째로 배치. 배치 op 1개 추가 수준.
- 회사 내부 배포의 킬러 기능 — 사내 표준 정의를 에이전트가 재사용하는 순간 도입 장벽이 낮아짐.
  (Raven Context Profiles와 같은 모양, 진입 장벽으로 검증을 걸 수 있는 게 우리 강점.)

## #4 재사용 템플릿 — 성공 작업의 typed 블록화

- 검증까지 통과한 배선 시퀀스/서브그래프를 입출력 계약과 함께 저장, 유사 요청 시 처음부터
  배선하는 대신 인스턴스화.
- **스코프 3계층 + 명시적 승격 (확정)**:
  - 프로젝트 단위(기본, LocalAppData 프로젝트 컨텍스트 폴더) → 개인 단위 → 회사 공유 라이브러리.
  - 공유 진입은 **의도적 승격 행위로만** — 조건: 검증 통과 + 출처 메타데이터(누가/언제/어떤 검증).
  - 자동 전파 금지: 템플릿은 실행되는 산출물이라 잘못된 템플릿은 오류를 조용히 복제함.
- #5 학습 파이프라인과 **별개 축**(명시·큐레이션 vs 암묵·통계)이되 상호 피드:
  로그 마이닝이 "반복 작업 → 템플릿 후보" / "템플릿 사용 후 사람 수정 빈발 → 강등 후보"를 제안.

## #5 사용 데이터 수집 파이프라인

- 수집 대상: 턴 전문(프롬프트+툴콜+잡 결과), 결과 라벨(커밋/Failed/거부), **암묵 신호**.
  - 핵심 암묵 신호: 에이전트가 만진 컴포넌트를 직후에 사람이 수동 수정 →
    **도메인별 fingerprint 변화로 공짜 탐지** → "결과 불만족" 라벨.
- 활용 순서 (파인튜닝은 마지막):
  1. 실사용 실패 사례 → 벤치 과제로 승격 (기존 벤치 루프에 실데이터 공급).
  2. 하우스룰 개선 — `assets/instructions/*.md`라 리빌드 없이 실험 가능(폴백 동기화 주의).
  3. (먼 미래) 파인튜닝 — 현 구독 CLI 백엔드로는 불가. API 경로/오픈 모델 전환 시 성공 세션의
     툴콜 시퀀스로 SFT.
- 회사 배포 실무: 사용 기록 고지 1장, **.3dm/.gh 파일 자체는 미수집**(이벤트/텍스트만),
  LocalAppData 컨텍스트 → 중앙 수집 옵션.

## #6 캔버스 정리 패스 — 레이아웃 + 그룹핑 + 시맨틱 명명 (2026-07-24 통합 확장)

시맨틱 명명 패스를 상위 기능으로 확장. 세 요소가 한 기능:
1. **레이아웃 정리** — 의존성 위상정렬 기반 계층 배치(소스 좌 → 결과 우, 와이어 교차 최소화).
   결정론적이므로 **서버 op가 좌표 계산**, 모델이 좌표를 찍지 않음.
2. **그룹핑** — workflow 단계별로 컴포넌트를 GH_Group으로 묶어 흐름이 보이게.
   그룹 경계를 어디서 자를지만 모델 판단.
3. **시맨틱 명명** — 그 그룹에 의미 이름("기둥 그리드 생성부", "파사드 패널 브랜치") →
   이후 지시가 결정적으로 바인딩.

- 두 모드: **생성 시 배치**(에이전트가 새 컴포넌트를 업스트림 기준 자동 위치 — 이게 더 중요,
  "GPTino가 만든 정의는 읽힌다"가 차별점. Raven 유저 불만 "WTH is that canvas" 참고) /
  **사후 전체 정리**(명시 요청 시만).
- human-wins 경계: 사람이 배치·수정한 영역은 기본 불가침, 전체 정리는 명시 요청일 때만.
- 기반 이미 랜딩됨: 도메인별 fingerprint 분리(3ca80b2, 이동=layout 해시만) 덕에 대량 재배치가
  값/배선 작업과 충돌하지 않음.
- 제약: **비파괴 방식만** (GH_Group은 컴포넌트를 안 건드림 — 정합). param nickname 강제 변경
  금지 — Raven이 nickname을 찍었다가 유저 반발로 버그 리포트 받은 사례.
